1키 컨셉

왜 만들었고, 무엇이 핵심 발상인가.

“코드베이스가 너무 커서 신입은 한 달 걸려 읽고, 시니어도 잊어버린다.” 이 위키는 raw 소스와 LLM이 쓴 설명 페이지를 1:1로 미러링하면서, 소스가 움직일 때마다 영향받는 페이지만 부분 갱신합니다. 사람은 골격을 정하고 LLM은 본문을 채웁니다.
①

1:1 미러링

raw/<top>/ ↔ wiki/raw/<top>/. 소스 파일 한 짝(.c+.h)에 페이지 하나. 길 잃지 않음.

②

사람은 골격, LLM은 본문

어떤 페이지가 존재할지는 사람이 결정 (또는 LLM 제안 → 사람 리뷰). 페이지 내용은 LLM이 생성·갱신.

③

Provenance 강제

모든 LLM 호출은 결과의 last_synced_sha, 모델, 시각, 본 파일을 front-matter에 남김. 누가/언제/어떤 sha에서 썼는지 추적 가능.

④

Diff 기반 patch-up

upstream이 움직이면 변경 파일만 라우팅 → 영향 페이지만 부분 갱신. 전면 재작성 금지.

⑤

Annealing

cron이 주기적으로 stale 페이지, 끊긴 링크, drift를 budget 상한 내에서 수리.

⑥

“신선해 보임” 안 만듦

출처가 안 움직였다 ≠ 결론이 맞다. 거짓 자신감 방지를 위해 auto-refresh 배지 의도적으로 배제.

코드 도메인에 맞춘 두 가지 디자인 결정

entity vs subsystem 분리

.c+.h 한 짝은 결정적으로 만들어진다 (seed_pages.sh). 여러 파일에 걸친 subsystem은 잘못 묶이면 그 위에 쌓이는 본문·sources·query 흔적이 모두 헛것이 되므로, LLM이 YAML 제안을 만들고 사람이 리뷰한 뒤 결정적으로 적용 (propose_layout → apply_layout).

coverage.json = 단일 진실

페이지 ↔ 소스 파일 매핑 + 신선도 ledger를 한 파일에 모음. 거의 모든 도구가 여기를 읽거나 씀. 라우팅 (diff → 어떤 페이지 갱신할지)도 여기로 결정. JSON 한 파일이라 git diff·리뷰 친화적.

5동작 예시 (실전 시나리오)

새 sub-tree 들이고 → 패치 사이클 한 바퀴 → 코드 리뷰 쿼리.

시나리오: raw/pcie_scsc/ 위키 만들기

일회 셋업 — LLM 백엔드 선택

config/llm.local.json에 default_profile을 claude 또는 ollama로 지정.

cp config/llm.example.json config/llm.local.json
${EDITOR:-vi} config/llm.local.json
python -m scripts.llm_client --probe        # 연결 검증

sub-tree 준비 — local git 보장

last_synced_sha는 sub-tree HEAD에서 가져옵니다. git이 아니면 sha는 null이 됩니다 (오류는 아님).

[ -d raw/pcie_scsc/.git ] || (cd raw/pcie_scsc && git init && \
    git add . && git commit -m "initial import")

Stub 일괄 생성 (entity)

.c+.h 짝마다 빈 페이지 + coverage.json 등록. idempotent.

bash scripts/seed_pages.sh --dry-run   # 미리보기
bash scripts/seed_pages.sh             # 실제 생성

(선택) Subsystem / concept layout 제안

여러 파일에 걸친 페이지. LLM이 YAML 제안 → 사람 리뷰 → 결정적 적용.

python -m scripts.propose_layout --tree raw/pcie_scsc --model claude-sonnet-4-5
${EDITOR:-vi} wiki/_meta/layout-proposals/pcie_scsc-*.yaml
python -m scripts.apply_layout wiki/_meta/layout-proposals/pcie_scsc-*.yaml

한 페이지 검증 후 batch

먼저 작은 페이지 한 장으로 워크플로 확인, 그 다음 일괄 시드.

# 단일 페이지
python -m scripts.update_wiki seed-agent \
    --page raw/pcie_scsc/osal/osal_helper.md \
    --model claude-sonnet-4-5

# 미채움 페이지 전체 batch
python -m scripts.seed_all --model claude-sonnet-4-5 --continue
bash scripts/seed_progress.sh   # 진행 상황

정적 사이트 빌드 + 열기

python -m scripts.build_site --clean
xdg-open site/raw/pcie_scsc/mlme.html
# 검색 기능 (file:// 제약 회피)
python -m scripts.build_site --serve --bind 127.0.0.1:8000

주간 패치 사이클 (cron)

upstream이 움직이면 영향 페이지만 부분 갱신.

# /etc/cron.d/code-llm-wiki
0 22 * * * cd /path/to/code-llm-wiki && \
  python -m scripts.sync_subtree --tree raw/pcie_scsc --record --out /tmp/m.json && \
  python -m scripts.patch_router --manifest /tmp/m.json --apply --out /tmp/r.json && \
  python -m scripts.update_wiki update --routing /tmp/r.json

주간 annealing — stale / 끊긴 링크 수리

python -m scripts.anneal scan                    # 후보 점검 (읽기만)
python -m scripts.anneal run --budget 3          # 상위 3개 수리

응용 — 패치 리뷰 쿼리 (grounding)

python -m scripts.update_wiki query \
    --template code-review \
    --input /tmp/patch.diff \
    --pages raw/pcie_scsc/mlme.md,raw/pcie_scsc/hip.md \
    --out wiki/queries/2026-05-15-mlme-review.md \
    --title "mlme: scan timing rework"

7.1 단기 (Quick Wins — 한두 PR이면 됨)

🪙

비용 / 토큰 대시보드

seed/update 호출마다 토큰·시간·모델을 _meta/cost.jsonl에 append. 주별 집계를 mkdocs 페이지로.

🔍

Diff 가시화

update 호출의 before/after 페이지 diff를 queries/_diff/에 보관. 리뷰어가 “LLM이 뭘 바꿨나” 한눈에.

🌐

한 페이지, 다국어 본문

front-matter에 lang: ko 추가, 같은 covers에 영문 버전. onboarding 자료로 활용.

📈

Incremental anneal scoring

매 사이클마다 전체 스캔 대신 cov 변경분만 재점수. 큰 위키일수록 효과 큼.

7.2 중기 (전략 과제 — 디자인 의사결정 필요)

🤖

GitHub PR 봇 통합

PR 열리면 자동으로 code-review 쿼리 실행 → 코멘트. 재사용 정책 강제 (PR마다 새로 실행).

🧭

임베딩 기반 의미 검색

현재 mkdocs 기본 검색은 키워드 기반. 페이지 벡터를 _meta/embeddings.jsonl에 캐시하고 cosine 검색 추가.

🧪

테스트 ↔ 페이지 매핑

covers[]에 테스트 파일도 포함. “이 함수의 테스트는 어디?” 를 페이지에서 바로.

🪞

multi-tree 의존성 그래프

sources[]를 cross-tree로 확장. raw/scsc-firmware가 raw/pcie_scsc를 참조하는 페이지를 묶음.

7.3 장기 (R&D — 검증 필요)

🗳️

multi-LLM voting / critique

두 모델이 같은 페이지를 시드 → 세 번째 모델이 reconciler. 비용 3배지만 중요 페이지에 한정 가능.

🕸️

Graph DB 백엔드

coverage.json 한 파일이 너무 커지면 graph DB로. 현재 규모(수백 페이지)에선 과한 최적화.

⏪

시간 여행 (sha별 페이지 스냅샷)

“6개월 전 mlme.c는 어떻게 설명됐었나” — 페이지 히스토리를 git tag로 자동 마킹. 포팅·디버깅에 유용.

🛡️

Adversarial 검증

LLM이 자기가 쓴 페이지를 다른 페이지·소스에 모순되는지 자동 점검. anneal의 한 카테고리로.

8팀이 자주 묻는 것

Q. 왜 위키를 LLM에 맡기나요? 사람이 쓰는 게 더 정확하지 않나요?

정확성은 그렇습니다. 문제는 유지보수성입니다. 코드는 매일 움직이는데 사람이 쓴 위키는 6개월이면 stale이 됩니다. 이 시스템은 “LLM이 처음 한 번은 덜 정확해도, 패치 사이클로 따라가는 비용이 사람보다 훨씬 낮다”는 가정에 베팅합니다. provenance를 강제해서 의심 시 추적 가능한 게 안전망입니다.

Q. LLM이 거짓말을 쓰면 어떻게 알아채나요?

완벽한 방어는 없습니다. 대신 세 가지 장치가 있습니다 — (1) sources[]에 LLM이 “내가 본 파일과 라인”을 자기 기록하므로 검증 가능, (2) code-review처럼 응용 쿼리는 재사용 금지 정책으로 “stale을 신선해 보이게” 만드는 함정 차단, (3) annealer가 끊긴 링크나 covers drift를 surface.

Q. 로컬 ollama로 정말 쓸만한가요?

페이지당 1시간 가까이 걸리고 품질은 Sonnet보다 떨어집니다. 야간 batch이나 비용 0이 절대 조건인 환경에 한정해서 권장. 본격 운영은 Sonnet 4.6 권장. API 인터페이스가 동일해서 코드 변경 없이 스위치 가능한 게 장점입니다.

Q. 다른 언어 / 프로젝트에도 쓸 수 있나요?

네. seed_pages.sh의 find 패턴만 손보면 .py, .go, .rs 등 어떤 언어든 동작합니다. 단 entity의 기본 단위인 “TU”가 언어마다 달라서 (Python은 모듈, Go는 패키지) layout 단계에서 조정이 필요합니다.
